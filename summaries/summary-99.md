# ECMAScript仕様輪読会 第99回 概要解説

今回のセッションでは、文字列操作に関する仕様の深い階層を確認し、サロゲートペアの扱いや、仕様書における「ジェネリック（汎用的）」なメソッドの定義の揺れについて議論が行われました。

## 1. String.raw (22.1.2.4)

`String.raw` は、テンプレートリテラルの「タグ関数」として使用されることを想定したメソッドです。バックスラッシュをエスケープシーケンスとして解釈せず、そのままの文字列として取得したい場合（Windowsのパスや正規表現の記述など）に重宝されます。

### アルゴリズムの要点
1.  **テンプレートオブジェクトの取得**: 第1引数から `raw` プロパティを取得し、オブジェクト化します。
2.  **リテラルと置換の結合**: テンプレート内の文字列部分（リテラル）と、`${}` で渡された置換部分（サブスティチューション）を交互に結合していきます。
3.  **エスケープの無視**: 通常のテンプレートリテラルでは `\n` は改行として「調理（cooked）」されますが、`raw` プロパティを参照することで、入力されたままの文字として扱います。

### コード例による検証
```javascript
// 改行コードがそのまま文字列として扱われる
console.log(String.raw`Hello\nWorld`); 
// 出力: "Hello\nWorld" （実際の改行は行われない）

// Windowsパスの記述に便利
const path = String.raw`C:\Users\Documents\Node`;
console.log(path); 
// 出力: "C:\Users\Documents\Node"
```

## 2. String.prototype オブジェクトの性質 (22.1.3)

`String.prototype` 自体の内部構造についても確認が行われました。

*   **String Exotic Object**: インデックスによるアクセス（`str[0]` など）を可能にするため、通常のオブジェクトとは異なる内部メソッド（`[[GetOwnProperty]]` など）を持つ「エキゾチックオブジェクト」として定義されています。
*   **内部スロット**: `[[StringData]]` スロットを持ち、初期値は空文字列です。
*   **length属性**: 文字列の長さを保持しますが、`writable: false`, `enumerable: false`, `configurable: false` であり、外部から書き換えることはできません。

---

## 3. String.prototype.at (22.1.3.1)

比較的新しく追加されたメソッドで、配列の `at` と同様の挙動を文字列でも実現します。

*   **負のインデックス**: `index < 0` の場合、`length + index` として扱われ、末尾からのアクセスが可能です。
*   **境界外の挙動**: 指定したインデックスが範囲外の場合、`undefined` を返します（後述の `charAt` との違い）。
*   **コードユニット単位**: あくまでUTF-16のコードユニット単位で処理するため、サロゲートペア（絵文字や一部の漢字など）を分割して扱う点に注意が必要です。

```javascript
const str = "𩸽"; // サロゲートペア（ホケ）
console.log(str.at(0));  // "\uD867" (上位サロゲート)
console.log(str.at(-1)); // "\uDE3D" (下位サロゲート)
console.log(str.at(10)); // undefined
```

## 4. charAt (22.1.3.2) と charCodeAt (22.1.3.3)

古くからあるメソッドの挙動を再確認しました。

*   **charAt(pos)**: 
    *   指定した位置の1コードユニットを文字列として返します。
    *   範囲外の場合は**空文字列 `""`** を返します（`at` は `undefined` なので対照的です）。
*   **charCodeAt(pos)**: 
    *   指定した位置のコードユニットを 0〜65535 の数値（16ビット整数）で返します。
    *   範囲外の場合は **`NaN`** を返します。

---

## 5. String.prototype.codePointAt (22.1.3.5)

Unicodeコードポイントを正しく扱うためのメソッドです。

*   **サロゲートペアの解釈**:
    *   指定した位置がサロゲートペアの開始（上位サロゲート）であれば、次のコードユニットと合わせて1つのコードポイントとしての数値を返します。
    *   指定した位置がサロゲートペアの後半（下位サロゲート）や、単独の文字であれば、その1要素分の値を返します。
*   **戻り値**: 範囲外の場合は `undefined` を返します。

```javascript
const str = "𩸽"; 
console.log(str.codePointAt(0)); // 171581 (0x29E3D)
console.log(str.codePointAt(1)); // 56893 (下位サロゲート単独の値)
```

---

## 6. String.prototype.concat (22.1.3.6)

文字列を連結するメソッドです。

*   **アルゴリズム**:
    1. `this` を文字列に変換。
    2. 引数として渡された各値を順に文字列に変換し、結合していく。
*   **パフォーマンス**: 現在のJavaScriptエンジンでは、`+` 演算子による連結の方が一般的かつ高速である場合が多いですが、仕様上の挙動はシンプルに「コードユニットのシーケンスの連結」です。

## 7. 仕様書の不備と「ジェネリック」に関する議論

今回の輪読会で最も盛り上がったのは、**メソッドが「ジェネリック（汎用的）」であるか否か**の記述についてでした。

*   通常、`String.prototype` のメソッドには「このメソッドは意図的にジェネリックに作られており、`this` が String オブジェクトである必要はない」という注釈（Note）がついていることが多いです。
*   しかし、今回読んだ `at` や `concat` の項目には、その注釈が欠落していました。
*   **実態の検証**:
    アルゴリズム内で `RequireObjectCoercible(this)` および `ToString(this)` が呼ばれているため、実際には数値やプレーンなオブジェクトに対しても実行可能です。
    ```javascript
    // String.prototype.charAt をオブジェクトに対して呼び出す例
    const obj = { toString() { return "ABC"; } };
    console.log(String.prototype.charAt.call(obj, 0)); // "A"
    ```
*   **結論**: 仕様書の記述漏れ（タイポやコピー＆ペースト時のミス）の可能性が高いという結論になり、コントリビューション（修正提案）の余地がある箇所として注目されました。

---

## まとめ

今回のセッションでは、文字列の基本操作メソッドを詳細に追い、以下の特性を再確認しました。
*   `at` は負のインデックスが使えるが、サロゲートペアは考慮しない（コードユニット単位）。
*   `codePointAt` を使うことで、Unicodeとしての正しい数値を扱える。
*   仕様書における「ジェネリック」の定義がメソッドによってまちまちであるが、内部アルゴリズム的には多くのメソッドが他の型に対しても動作するように設計されている。
